# 🛡️ 安全响应与策略 (Security Policy)

非常感谢您关注 **Snippet Collector** 的安全性。由于本项目涉及多环境较高权限的调用（如无障碍权限、系统全局热键探测、日志监听、悬浮窗拦截），安全性和用户隐私防护始终是社区架构维护组的第一指导要旨。

## 受支持的版本范围

目前，只有 `main` 主分支上的最新功能特性受到积极的安全更新及补丁支持。建议所有部署实例均跟随最新发布版节点运行。

## 发现后门或漏洞应该如何报告？ (Reporting a Vulnerability)

如果您在代码框架本身或脚本周边依赖库中发现了可能被恶意利用的提权漏洞、命令执行注入或是涉及到系统隐私与安全敏感数据的非常规调用泄露等严重情形：

**🚨 请绝对不要**通过公开可见的 GitHub Issue 页面或讨论区进行报告，以防止此项漏洞细节在被修复前曝光遭受更广泛的恶意利用影响。

请优先使用 GitHub 提供的【私密漏洞报告机制】，这是当前最安全、可规范追踪的协作渠道。若您因故无法使用此功能，也可通过受保护的电子邮件隐秘联络我们。

🔒 **优先通道：GitHub Private Vulnerability Reporting**  
请转至本仓库顶部导航栏，依次进入 **Security** 选项卡 -> **Advisories** -> 点击 **Report a vulnerability** 按钮进行提报。

📧 **备用邮箱通道**： `[请在此处替换为您的安全接收邮箱 -> 例如：security@snippetcollector.team]`

我们将在收到您的汇报流程执行方案后：
1. 立即响应，在最迟 **48 小时**内确认并立档分析报告内容。
2. 架构组将分配最高级别的资源展开排查，并在具有可行措施的最短时间规划全平台修复更新发布包。
3. 若高危漏洞报告被查实得到修正后，如果在遵循此保密规范的前提下，我们将会以极其诚挚的感激身份在发行版（Release Note）与维护圈中对您的贡献表示表彰和感谢（当然也能无条件尊重及保护您隐蔽和匿名的意愿）。

**请千万不要在任何公开的渠道发表相关的可复现分析猜想或漏洞概念验证代码 (PoC)，甚至连提问交流也应避免进行，直到对应的严重缺陷获得全链条修复。**非常感激所有的白帽和安全开发者共同协助我们守护开源社区安全！
